## 1. Scope and Method

- **Inventory**: 20 destination files (11 ONNX + 7 inference.yml + 2 PNG)

- **Audit date**: 2026-07-04

## 2. Source Topology

### Upstream distribution pattern

The upstream BetterGI project distributes model files **bundled inside the Windows installer executable** (e.g., `BetterGI.Install.0.62.0.exe`), available via GitHub Releases. Models are **not** committed to the git repository and **not** published as a standalone ONNX archive.

Key observations from upstream publish workflow:
- `dotnet publish` builds the application
- Model files are included in the output directory before packaging
- The `babalae/bettergi-publish` repo contains only map data (71 MB `AllMap.zip`), not model files
- Model files originate from PaddleOCR's pre-trained models, converted to ONNX format

### Topology verdict: **Single archive (conversion required)**

All 20 destination files originate from the upstream BetterGI's release distribution. However:
- The upstream provides these files only inside the Windows installer (`.exe`)
- There is **no standalone, immutable ONNX archive URL** published by the upstream
- Model ONNX files originate from PaddleOCR pre-trained inference models, converted via Paddle2ONNX
- The exact version/tag of upstream's bundled models is not independently published

**Status**: `Unresolved` — no immutable source archive URL available from upstream

## 4. Source Candidates

### Candidate A: Upstream BetterGI Release (Recommended, blocked)

- Source: `https://github.com/babalae/better-genshin-impact/releases`
- Format: Installer `.exe` (Windows)
- Contains all 20 files bundled
- **Problem**: No standalone archive URL; requires extracting from installer
- **Problem**: No SHA-256 for individual model files published
- **Status**: `Unresolved`

### Candidate B: monkt/paddleocr-onnx (HuggingFace)

- Source: `https://huggingface.co/monkt/paddleocr-onnx`
- Format: HuggingFace model repo with individual files
- Has PP-OCRv5 models for multiple languages
- **Problem**: Uses external `dict.txt` files (`character_dict_path`), NOT inline YAML `character_dict`
- **Problem**: Directory structure does NOT match manifest (uses `detection/v5/det.onnx`, `languages/english/rec.onnx`)
- **Problem**: Version pinned to specific PP-OCR release, not BetterGI's version
- **Status**: `Blocked` — incompatible character_dict format and directory layout

### Candidate C: PaddlePaddle/PaddleOCR official

- Source: `https://github.com/PaddlePaddle/PaddleOCR`
- License: Apache 2.0 for model weights
- Format: Paddle inference model (`.pdmodel` + `.pdiparams`), NOT ONNX
- **Problem**: Requires Paddle2ONNX conversion — not direct download
- **Problem**: Conversion pipeline must be reproducible (opset, input/output names, dynamic shapes)
- **Status**: `Blocked` — conversion required, no pre-converted ONNX

### Candidate D: kreuzberg-dev/paddle-to-onnx (GitHub)

- Source: `https://github.com/kreuzberg-dev/paddle-to-onnx`
- License: Apache 2.0
- Format: GitHub Release ONNX assets
- Has PP-OCRv5 detection + recognition ONNX
- **Problem**: Model versions may not match upstream BetterGI's exact pick
- **Problem**: Directory layout does NOT match manifest
- **Status**: `Unresolved` — version mismatch risk; upstream BetterGI may use different model picks

### Candidate E: jingsongliujing/OnnxOCR (GitHub)

- Source: `https://github.com/jingsongliujing/OnnxOCR`
- Has `download_models.py` helper
- **Problem**: Third-party project, not upstream BetterGI
- **Problem**: Model versions unknown
- **Status**: `Unresolved`

## 8. inference.yml Compatibility

The manifest contract requires: inline `PostProcess.character_dict` as YAML sequence.

From upstream `PaddleOcrService.cs`:
```csharp
if (parser.Current is not YamlDotNet.Core.Events.Scalar { Value: "character_dict" }) continue;
// Reads scalars from the sequence directly
```

**Must NOT use `character_dict_path`** or external `dict.txt` files. The `monkt/paddleocr-onnx` source uses `dict.txt` — incompatible with current parser.

**Status**: Compatibility is presumed (upstream produces inline character_dict inference.yml), but cannot be verified without examining real files.

- When source-locking, must either:
  - Use upstream's pre-converted ONNX (if upstream provides as release asset), or
  - Create a reproducible Paddle2ONNX conversion pipeline and document output checksum
